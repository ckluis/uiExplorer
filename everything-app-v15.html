Proposal: Migrate to event-driven architecture
MT
Marcus Torres 2h ago Hot

I'd like to propose that we migrate our core platform to an event-driven architecture. After analyzing our current system, I believe this would solve several pain points we've been hitting as we scale.

Key benefits:

  • Decoupled services — Each module (CRM, Projects, Wiki) can evolve independently without cascading deployments
  • Real-time processing — Events propagate instantly across the system, enabling features like live dashboards and cross-module notifications
  • Fault tolerance — If one service goes down, events queue up and replay when it recovers. No data loss.
  • Audit trail — Every state change is an event, giving us a complete, immutable history for compliance and debugging

For the message broker, I'm leaning toward NATS for its simplicity and performance, but I'm open to Kafka if we need stronger ordering guarantees. Would love to hear everyone's thoughts on the trade-offs.

I've drafted an RFC document in the Wiki with more technical details including migration phases, estimated timeline (3 sprints), and rollback strategy.

API Reference v3.2
Last updated 2h ago by Marcus Torres | Contributors:
MT AI
Up to date v3.2 • 47 endpoints • REST
Overview

The NovaCorp API v3.2 provides programmatic access to all platform resources including contacts, deals, analytics, and document management. All endpoints use JSON request and response bodies. The base URL for all API calls is https://api.novacorp.io/v3.

Authentication is required for all endpoints. Rate limits are enforced per API key. See the Authentication and Rate Limits sections below for details.

Endpoints
Method Path Description Auth
GET /api/v3/contacts List all contacts Bearer token
POST /api/v3/contacts Create contact Bearer token
GET /api/v3/deals List deals Bearer token
PUT /api/v3/deals/:id Update deal Bearer token
GET /api/v3/analytics/pipeline Pipeline analytics Admin
POST /api/v3/receipts/scan Upload receipt scan Bearer token
Authentication

All API requests must include a valid Bearer token in the Authorization header. Tokens are generated from the Settings > API Keys page and are scoped per workspace.

Authorization Header curl
curl -X GET https://api.novacorp.io/v3/contacts \ -H "Authorization: Bearer YOUR_API_KEY" \ -H "Content-Type: application/json" # Response { "data": [ { "id": "ct_8a4f2e", "name": "Alex Rivera", "company": "Acme Corp", "email": "alex@acmecorp.com" } ], "meta": { "total": 142, "page": 1 } }
Rate Limits

API requests are rate limited to 1,000 requests per minute per API key for standard endpoints. Analytics endpoints are limited to 100 requests per minute. Rate limit headers are included in every response:

Rate Limit Headers
X-RateLimit-Limit: 1000 X-RateLimit-Remaining: 847 X-RateLimit-Reset: 1707580800

When the rate limit is exceeded, the API returns a 429 Too Many Requests response. Implement exponential backoff in your client to handle this gracefully.
